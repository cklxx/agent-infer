## Why kernel needs scale_perm_single on s2(s_channel)

Kernel `marlin_w4a8_kernel.cu` reads two scale tensors during dequant:
- `s2` = per-channel scale(`s_channel`)
- `s3` = per-group scale(`s_group_stored = s_group / s_channel`)

Both s2 and s3 must follow the same per-thread fragment layout used by
mma `m16n8k16.row.col.s32.s8.s8.s32`。`scale_perm_single` shuffles
s_channel into the layout the kernel expects(specifically the per-channel
N-dimension layout where each thread reads channels at strided positions
matching its B fragment lanes)。

Without permutation,kernel reads s_channel from **wrong N positions** for
each thread → multiplies dequantized weight by wrong channel scale →
**every output element gets a wrong scale factor** → semantic gibberish
(matches observed:per-thread fragment ordering OK from H3 row-fix,but
cross-channel scale mismatch produces multilingual real-word jumble)。
