## Bug in `2fafa9e` recipe at Step 2(lines 81-88)

```fish
# WRONG — profiles python bench client, not server
nsys profile \
    --output /tmp/p0.0-phase1a-multitenant \
    --trace cuda,nvtx,osrt \
    --duration 30 \
    --capture-range=none \
    /home/ckl/projects/arle/.venv/bin/python \
    scripts/bench_multitenant_burst.py \
    http://localhost:8000 Qwen/Qwen3-4B
```

**Why wrong**:`nsys profile <cmd>` profiles `<cmd>` and its child processes。
Here `<cmd>` is the python bench client。Server is started SEPARATELY in
background → server is NOT a child of the profiled process → server NVTX
scopes(`step_admission_prefix_lookup` from `5a63142`)are **not captured**。

`nvtxsum` report would show empty/zero server-side NVTX entries。Step 4
4-phase decomposition data would be missing → entire P0.0 Phase 1.A bench
yields no usable signal。

## Two correct patterns

### Option A — profile server directly(recommended)

```fish
# Profile the server process; bench client runs separately
nsys profile \
    --output /tmp/p0.0-phase1a-multitenant \
    --trace cuda,nvtx,osrt \
    --duration 30 \
    ./target/release/infer \
        --model-path infer/models/Qwen3-4B \
        --port 8000 \
        --num-slots 8 \
        --max-seq-len 8192 \
        --admission-policy prefix-aware &
NSYS_PID=$!
sleep 5

# Bench client runs in foreground, generating workload
/home/ckl/projects/arle/.venv/bin/python \
    scripts/bench_multitenant_burst.py \
    http://localhost:8000 Qwen/Qwen3-4B

# nsys --duration 30 will auto-stop after 30s; wait for trace finalize
wait $NSYS_PID 2>/dev/null
```

**Pros**:simple,nsys captures server start-to-end,automatic shutdown after
30s。

**Cons**:server warmup(~1.25s per dfa6408)is included in trace。But this is
fine because warmup runs ONCE,bench burst dominates the 30s window。

### Option B — attach nsys to running server

```fish
# Start server first
./target/release/infer \
    --model-path infer/models/Qwen3-4B \
    --port 8000 \
    --num-slots 8 \
    --max-seq-len 8192 \
    --admission-policy prefix-aware &
SERVER_PID=$!
sleep 5

# Wait for server fully ready, then attach nsys
nsys profile \
    --output /tmp/p0.0-phase1a-multitenant \
    --trace cuda,nvtx,osrt \
    --duration 30 \
    --attach $SERVER_PID &

# Bench client generates workload
/home/ckl/projects/arle/.venv/bin/python \
    scripts/bench_multitenant_burst.py \
    http://localhost:8000 Qwen/Qwen3-4B

kill $SERVER_PID
```

**Pros**:server warmup excluded from trace。

**Cons**:more complex,`--attach` requires CAP_SYS_ADMIN or `setcap`
capability on nsys binary(may need sudo),timing sensitive(must attach
between server-ready and bench-start)。

→ **Recommendation**:Option A。Simpler,dominates by bench window,no perm
requirements。Server warmup represents <5% of 30s trace if bench is
sustained。

## Why I missed this originally

When I wrote `2fafa9e` recipe at EOD+85,I conceptualized "nsys profile a
bench" without distinguishing server-side NVTX emission from client-side
profiling target。Mental model defaulted to "nsys profile <bench command>"
without thinking about which process emits the scopes I want to measure。

§0 SOLID rule application:**before writing a recipe,trace the data flow**:
1. WHAT data am I measuring?(server-emitted NVTX scopes)
2. WHERE does that data come from?(server process)
3. HOW does my profiling tool reach that data?(profile target = source of
   data,not consumer)

Skipped step 3 in eod85 → recipe bug。Audit-of-audit step (b55bfcd) +
this brief together cover the missed step 3 retroactively。
