## §0 SOLID gap #2 — cublasLt vs TileLang routing for prefill GEMM unverified

`warmup.rs:103` docstring:
> Pass 1 drives `forward_decode_batch` per batch size, which populates the
> cublasLt heuristic algo cache for every shape

→ Decode GEMMs go through cublasLt and benefit from autotune cache。

**But**:does prefill GEMM use the same cublasLt path?Per CLAUDE.md
"TileLang drives CUDA paged prefill/decode for BF16 attention"。Prefill
**attention** = TileLang AOT cubins(pre-built,no warmup needed)。Prefill
**GEMM**(qkv_proj,o_proj,mlp,gate_up,down)= unknown routing。

If prefill GEMM also goes through cublasLt → P0.3 prefill warmup pays off
the same way decode does(populate algo cache for new M = prompt_length
shapes)。

If prefill GEMM goes through TileLang AOT cubins → cubins are already
built at compile time,no runtime warmup possible/needed for them,and
P0.3 prefill warmup buys ~zero。

**Resolution**:single grep at `infer/src/ops/` for prefill GEMM dispatch
will confirm。(Out of scope for this brief — P0.3 codex will do that
naturally as part of implementation。)

→ §0 SOLID rule:**don't commit to LOC estimate until routing question is
answered**。If prefill GEMM is TileLang-only,P0.3 axis is wrong and
bimodal root-cause must be re-investigated。
